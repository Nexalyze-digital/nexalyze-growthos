# GrowthOS v0.5.1 Post-Mortem

## Release Summary

GrowthOS v0.5.1 hardened the v0.5.0 platform foundation before v0.6.0 feature work. The release validated PostgreSQL locally, rehearsed backup and restore, improved frontend session handling, added role-aware UI behavior, and prepared CI-ready Playwright smoke coverage.

The release intentionally avoided Publishing Engine work. Its purpose was to reduce platform risk before the next product capability.

## PostgreSQL Validation Results

- PostgreSQL version validated: 17.10.
- Validation database: `growthos_v051_validation`.
- Restore rehearsal database: `growthos_v051_restore`.
- Alembic upgrade from empty database: passed.
- Alembic downgrade to base: passed.
- Alembic re-upgrade to head: passed.
- Backend test suite against PostgreSQL: 50 passed.
- JSON-to-PostgreSQL migration and duplicate protection: passed through PostgreSQL-backed tests.

The validation surfaced a useful portability issue: SQLite allowed fixture inserts that PostgreSQL rejected because of immediate foreign-key enforcement. The fixture was hardened by explicitly flushing parent rows before child rows.

## Backup And Restore Lessons

- Backup and restore should be rehearsed on disposable databases before any production migration.
- Custom-format `pg_dump` output is appropriate for controlled restore rehearsal.
- Restore is inherently destructive when using `--clean --if-exists`, so production use requires explicit operator approval and verified backups.
- Helper scripts should avoid echoing full database URLs because URLs may contain credentials in real deployments.
- Restore verification should query expected tables after the restore, not merely trust a successful command exit.

## Authentication And Session Improvements

- Frontend requests now refresh expired access tokens before authenticated API calls.
- A 401 response triggers one refresh and one retry of the original request.
- Concurrent refresh attempts share a single refresh promise.
- Failed refresh clears the stored session and returns the user to the login gate.
- Workspace selection is preserved after successful refresh when the refreshed session still includes the selected workspace.
- Session restoration was adjusted to avoid hydration mismatch during client startup.

## Security Improvements

- PostgreSQL configuration now uses `psycopg` and SQLAlchemy pool pre-ping for safer long-lived connections.
- Role-aware UI hides viewer write controls while preserving backend authorization as the source of truth.
- Token-storage risks are documented with a recommended HttpOnly-cookie migration path.
- Restore logging no longer echoes full database URLs.
- CI smoke coverage uses mock provider settings and does not require Ollama or secrets.
- Secret scanning and forbidden-artifact review remained clean.

## Remaining Technical Debt

- Hosted production PostgreSQL validation remains required.
- Tokens remain in browser `localStorage`; HttpOnly-cookie migration is still recommended.
- Rate limiting remains in-memory and is not horizontally scalable.
- No MFA, email verification, password reset, invite flow, or admin portal exists yet.
- Playwright workflow should be observed in GitHub Actions before becoming a required check.
- A moderate PostCSS advisory remains through Next.js and should be resolved through a compatible framework upgrade path.

## Recommendations Before v0.6.0

- Decide whether hosted PostgreSQL validation should be completed before or alongside Publishing Engine work.
- Keep Publishing Engine scoped to draft creation, approval state, and workspace ownership before adding external social publishing connectors.
- Add role-aware tests for publishing workflows from the beginning.
- Avoid storing publishable drafts outside the database-backed workspace model.
- Continue treating backend authorization as authoritative, with frontend controls as a usability layer.
- Track the Next.js/PostCSS advisory and plan a compatible framework upgrade when safe.
