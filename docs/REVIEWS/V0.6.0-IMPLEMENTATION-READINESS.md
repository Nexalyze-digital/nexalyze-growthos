# v0.6.0 Publishing Engine Implementation Readiness Review

## Executive Summary

The v0.6.0 Publishing Engine planning package is ready for implementation after documentation refinements made in this review. The architecture, database model, API contract, user flows, security review, QA strategy, and release scope now align around a controlled foundation release: workspace-scoped drafts, review, scheduling, queueing, and deterministic mock publishing.

No application code should be written until the planning PR is reviewed and accepted. The next work should be split into independent implementation PRs that preserve existing AI Content Studio, Brand Brain, Research Hub, authentication, workspace isolation, and PostgreSQL behavior.

Readiness score: 92/100.

## Architecture Review

The proposed architecture follows the established GrowthOS backend pattern: authenticated API routes, Pydantic schemas, request context, role enforcement, service layer, repository layer, SQLAlchemy persistence, and provider abstraction.

Strengths:

- Keeps Publishing Engine separate from Content Studio while allowing draft handoff.
- Preserves Brand Brain and Research Hub as upstream context sources rather than coupling them into publishing persistence.
- Uses backend-authoritative workspace and role checks.
- Defers real social OAuth and provider posting until official credentials and security storage are ready.
- Prohibits browser automation for social posting.

Refinement applied:

- Clarified that scheduling and queue jobs pin `draft_version_id` so later edits cannot silently alter approved or scheduled content.

Architecture score: 94/100.

## Data Model Review

The model includes the required entities for v0.6.0:

- `drafts`
- `draft_versions`
- `approvals`
- `approval_comments`
- `schedules`
- `publishing_jobs`
- `publishing_attempts`
- `social_connections`
- `workspace_publishing_settings`

Relationships align with workspace isolation and PostgreSQL migration readiness. Drafts can optionally reference Brand Brain and Research Hub records without forcing those modules to change.

Refinements applied:

- Added `draft_version_id` to `schedules`.
- Added `draft_version_id` to `publishing_jobs`.
- Clarified that archive is the primary user-facing removal flow.
- Clarified that soft delete is protected cleanup and must preserve audit/history records.

Remaining future considerations:

- A later release should add OAuth token storage with encryption/key management.
- A later release may need provider account/page/channel entities once real social integrations begin.
- Multi-step approval policy templates are intentionally deferred.

Database score: 91/100.

## API Review

The API contract covers draft CRUD, version history, review, scheduling, queue management, publishing history, and connection status.

Strengths:

- Resource grouping is clear under `/api/v1/publishing`.
- Error semantics are aligned with existing API conventions.
- Write endpoints are explicitly role-protected.
- Calendar and queue APIs are separated from draft lifecycle APIs.

Refinements applied:

- Clarified `DELETE /drafts/{draft_id}` as a reserved/protected cleanup endpoint, with archive/restore remaining the primary user-facing workflow.
- Clarified version pinning behavior on schedule and enqueue endpoints.

Missing endpoint candidates for implementation review:

- `GET /settings` and `PUT /settings` for workspace publishing settings should be added to the API contract during implementation.
- `GET /review-queue` may be useful for admin/owner review dashboards, even if it is implemented as filtered draft listing.
- `POST /jobs/{job_id}/process` should remain internal-only if added for local mock processing.

API score: 90/100.

## Security Review

The security plan correctly treats frontend controls as advisory and backend permissions as authoritative.

Required safeguards:

- Enforce workspace isolation for every draft, approval, schedule, job, attempt, setting, and connection query.
- Escape/render draft content as text only.
- Enforce admin/owner approval, scheduling, retry, and cancel rights.
- Enforce self-approval prevention when configured.
- Pin approved draft versions before scheduling and publishing.
- Store only safe provider summaries.
- Store no social tokens in v0.6.0.

Known platform debt remains outside v0.6.0:

- Access tokens are still stored in localStorage.
- Rate limiting remains in memory.
- MFA, email verification, password reset, invite flow, and admin portal remain planned.

Security score: 92/100.

## UX Review

The UX plan covers the expected operational surfaces:

- Content Library
- Draft Detail
- Draft Editor
- Review Queue
- Publishing Calendar
- Publishing Queue
- Publishing Settings
- Social Connection Status

Strengths:

- Role-aware controls are included but do not replace backend checks.
- Empty, loading, error, validation, mobile, and accessibility states are documented.
- Calendar and queue views are planned as work surfaces rather than marketing pages.

Suggested refinement for implementation:

- Keep the first release dense and operational: tables, filters, drawers/dialogs, and compact forms.
- Avoid overbuilding drag-and-drop scheduling until the baseline calendar and reschedule workflows are stable.

UX score: 89/100.

## Testing Review

The QA plan covers backend unit tests, migration validation, frontend validation, Playwright smoke, role workflows, and regression checks.

Required additions during implementation:

- Explicit tests for version-pinned schedule and queue behavior.
- Cross-workspace negative tests for every nested resource.
- Regression checks for Content Studio save-to-draft handoff.
- Mock adapter tests for idempotency collision, transient retry, permanent failure, and safe error summaries.

Testing score: 91/100.

## Dependency Review

The plan appropriately avoids new dependencies during planning. Implementation should use existing stack defaults first:

- Next.js and React for frontend.
- Existing local components and styling conventions.
- FastAPI, Pydantic, SQLAlchemy, and Alembic for backend.
- Python `zoneinfo` for timezone validation.
- Playwright for browser smoke coverage if already present in the project.

No dependency should be added for calendar UI, queues, or external social publishing unless implementation proves the existing stack is insufficient and the dependency passes license/security review.

## Open Source Review

`OPEN-SOURCE-CATALOG.md` and `ADOPTION-REGISTER.md` were reviewed. The Publishing Engine plan adopts only architectural patterns:

- n8n: job metadata, attempt history, retry visibility, and workflow monitoring concepts only.
- Playwright: role-based browser validation patterns only.
- Radix UI: accessible component behavior expectations only.
- Next.js: existing App Router/authenticated shell composition patterns only.

No third-party source code is approved for copying into GrowthOS from these repositories.

## Risks

- Scope creep into real social OAuth or live posting.
- Publishing stale draft content if version pinning is missed.
- Duplicate publishes if idempotency is not enforced transactionally.
- Timezone conversion mistakes.
- Approval policy ambiguity.
- Mobile calendar complexity.
- Stored XSS if draft text is rendered unsafely.

## Missing Requirements

The planning package is sufficient to begin implementation, but these details should be resolved in the first implementation PR:

- Add explicit settings API endpoints.
- Define platform enum values and per-platform content constraints.
- Decide whether soft delete ships in v0.6.0 or remains reserved.
- Define default approval setting behavior for newly created workspaces.
- Define the exact Content Studio save-to-draft handoff payload.
- Define audit event names for publishing workflows.

## Suggested Refinements

Applied in this review:

- Version pinning added to data model, API contract, scope, QA, and implementation plan.
- Soft delete clarified as protected cleanup rather than primary UX.
- Implementation package sequence clarified for independent delivery.

Suggested for implementation PRs:

- Start with backend foundation and migration before exposing frontend navigation.
- Keep real social provider code behind future interfaces only.
- Keep mock processing deterministic and fast.
- Add settings endpoints with owner-only write access.

## Implementation Order Validation

The proposed packages can be delivered independently without breaking existing functionality if each PR keeps new publishing routes isolated and protected.

Recommended order:

1. Database models, Alembic migration, schemas, repositories, settings defaults.
2. Draft CRUD, versioning, archive/restore, duplicate, Content Studio draft handoff.
3. Review workflow, lifecycle policy, approval comments, role enforcement.
4. Scheduling, timezone conversion, version pinning, conflict validation.
5. Publishing queue, attempt history, idempotency, retry policy, mock adapter.
6. Frontend Content Library, Draft Detail, Draft Editor, Review Queue.
7. Calendar, Queue, Settings, Connection Status, mobile and accessibility pass.
8. End-to-end validation, regression hardening, QA docs, release readiness.

Recommended number of implementation PRs: 6 to 8.

## Definition Of Done For Each Implementation Package

### Package 1: Persistence Foundation

- Publishing tables and indexes exist.
- Alembic upgrade and downgrade pass on SQLite and PostgreSQL.
- Repositories enforce workspace filtering.
- No frontend navigation is exposed yet.

### Package 2: Drafts

- Editors/admins/owners can create, edit, duplicate, archive, and restore drafts.
- Viewers remain read-only.
- Draft versions are created deterministically.
- Content Studio handoff creates a workspace-scoped draft.

### Package 3: Review Workflow

- Editors can submit drafts.
- Admins/owners can approve, reject, and request revision.
- Self-approval prevention works when enabled.
- Review history and comments are durable.

### Package 4: Scheduling

- Only approved drafts can be scheduled.
- Schedule records pin approved draft versions.
- Timezone conversion is tested.
- Conflict detection returns `409`.

### Package 5: Queue And Mock Provider

- Queue jobs pin draft versions.
- Idempotency prevents duplicates.
- Mock adapter supports success, transient failure, permanent failure, retry, and safe summaries.
- Attempt history is durable.

### Package 6: Frontend Core

- Content Library, Draft Detail, Draft Editor, and Review Queue work for all roles.
- Empty, loading, validation, error, and offline states are visible.
- Existing modules do not regress.

### Package 7: Calendar And Operations UI

- Calendar, Queue, Settings, and Connection Status screens work on desktop and mobile.
- Owner-only settings writes are enforced.
- Keyboard navigation and focus behavior are acceptable.

### Package 8: Release Hardening

- Backend tests pass.
- PostgreSQL migration validation passes.
- Frontend lint and build pass.
- Playwright smoke passes.
- Secret scan and forbidden artifact review pass.
- Release notes and QA docs are updated.

## Recommendation

Ready after documentation updates. The updates in this review are documentation-only and close the main readiness gaps. After the planning PR is reviewed, v0.6.0 implementation can begin without redesign.
