# v0.6.0 Publishing Engine Implementation Plan

## Executive Summary

v0.6.0 should deliver the first controlled Publishing Engine foundation for GrowthOS. The release should persist drafts, manage review and approval, support scheduling metadata, expose a publishing queue, and validate all workflows with a deterministic mock publishing adapter.

This release must not implement real social publishing unless official credentials and API access are available. It must not use browser automation to bypass platform APIs or security controls.

## User Stories

- As an editor, I can save generated Content Studio output as a workspace draft.
- As an editor, I can edit, duplicate, archive, restore, and submit drafts for review.
- As an admin, I can approve, reject, request revision, schedule, retry, and cancel publishing jobs.
- As an owner, I can configure workspace publishing settings and approval requirements.
- As a viewer, I can read drafts, calendar items, and queue status without write controls.
- As a reviewer, I can see approval history, comments, revisions, and audit context.

## Scope

- Workspace-scoped draft persistence.
- Draft lifecycle and version history.
- Review workflow with approval history and reviewer comments.
- Content Library with search, filters, sort, pagination, and archived view.
- Calendar foundation with month, week, day, timezone-aware scheduling, conflicts, reschedule, and unschedule.
- Publishing queue foundation with retry and idempotency.
- Deterministic mock publishing adapter for LinkedIn, X, Facebook, and Instagram.
- External adapter interfaces only.
- Backend tests, frontend validation, Playwright smoke, and documentation.

## Out Of Scope

- Real OAuth integrations.
- Browser-based social posting.
- Paid social APIs.
- Billing.

- Background infrastructure beyond the queue foundation required by Publishing Engine.

## Architecture

Publishing should follow the existing GrowthOS pattern:

1. Authenticated frontend calls `/api/v1/publishing/*` with access token and active workspace id.
2. API routes validate payloads with Pydantic.
3. `RequestContext` enforces workspace membership.
4. Role-specific dependencies or service checks enforce lifecycle permissions.
5. Publishing services coordinate repositories, transition policy, scheduler, queue, mock adapter, and audit events.
6. SQLAlchemy repositories persist normalized publishing tables.
7. Frontend renders role-aware controls while backend remains authoritative.

## State Transition Diagram

```text
generated -> edited
generated -> ready_for_review
edited -> ready_for_review
ready_for_review -> approved
ready_for_review -> rejected
rejected -> edited
approved -> scheduled
scheduled -> publishing
publishing -> published
publishing -> failed
failed -> scheduled
failed -> archived
generated|edited|ready_for_review|approved|rejected|scheduled -> archived
archived -> edited
```

Invalid transitions should return `409 Conflict`.

## Queue And Retry Model

- Queue job is created when an approved draft is enqueued for publishing.
- Jobs use deterministic idempotency keys: `workspace_id:draft_id:platform:scheduled_at:version`.
- Job states: pending, processing, succeeded, failed, cancelled.
- Attempts record retry count, provider status, error category, non-sensitive response summary, and timestamp.
- Transient mock failure allows retry with `next_retry_at`.
- Permanent mock failure blocks retry unless the draft is edited or requeued manually.

## Scheduling Model

- Store `scheduled_at_utc` plus `workspace_timezone`.
- Validate input timezone with Python standard zoneinfo.
- Reject past schedule times.
- Detect conflicts by workspace, platform, and schedule window.
- Month/week/day queries return normalized event records.
- Reschedule updates schedule and related queue metadata in one transaction.

## Mock Adapter Design

The deterministic mock adapter should support:

- LinkedIn, X, Facebook, Instagram.
- Successful publish.
- Transient failure.
- Permanent failure.
- Retry.
- Idempotency.
- Provider post id.
- Published timestamp.

No network calls are made.

## External Adapter Boundaries

Real adapters should remain interfaces only in v0.6.0:

- `PublishingProvider`
- `SocialConnectionProvider`
- `PublishResult`
- `PublishError`

Adapters must not log credentials, tokens, or raw provider responses.

## Migration Plan

1. Add SQLAlchemy models and Alembic migration.
2. Create tables with foreign keys, indexes, and constraints.
3. Add workspace settings defaults.
4. Backfill nothing in v0.6.0 unless draft import is explicitly added.
5. Validate SQLite and PostgreSQL migrations.
6. Add rollback migration that drops publishing tables in dependency order.

## Rollback Plan

- Stop API writes.
- Back up PostgreSQL.
- Downgrade Alembic if no production publishing data must be preserved.
- If data exists, restore backup rather than destructive downgrade.
- Disable frontend Publishing navigation if backend rollback is required.

## Definition Of Done

- Drafts are workspace-scoped and role-protected.
- Lifecycle transitions are enforced and tested.
- Approval history and audit events are durable.
- Scheduling is timezone-aware and conflict-checked.
- Mock publishing succeeds, fails transiently, fails permanently, retries, and respects idempotency.
- Frontend supports desktop and mobile workflows.
- Existing Content Studio, Brand Brain, Research Hub, auth, and workspace flows do not regress.

## Release Risks

- Scope creep into real social APIs.
- Timezone edge cases.
- Approval policy ambiguity.
- Queue idempotency mistakes.
- Draft version mismatch during approval or publishing.
- UI complexity on mobile.

## Merge Strategy

- Use small implementation PRs behind visible but controlled navigation.
- Keep each backend package independently testable.
- Merge database and API foundation before full frontend workflows.
- Require Playwright smoke before release candidate.

## Recommended Implementation Order

1. Data model and migration.
2. Draft service/repository/API.
3. Lifecycle policy and role tests.
4. Review workflow.
5. Scheduling.
6. Queue and mock adapter.
7. Frontend workflows.
8. Regression, QA, and release docs.
